## 1. The product in one screen

**rec** is a personal, single-household house-search intelligence portal. It is not a generic SaaS —
it is one buyer's command centre. It does six things:

| Capability | Where it lives | One-line purpose |
|---|---|---|
| **Dashboard** | `index.html` + `assets/js/dashboard/` | At-a-glance verdict on readiness, money, and next best action. |
| **Finances** | `pages/finances.html` + `assets/js/finances/` | Affordability, deposit, LISA, money-flow, savings velocity, investments. |
| **Areas** | `pages/areas.html`, `area-detail.html`, `map.html` | Researched location dossiers + matching + map. |
| **Listings** | `pages/listings.html`, `property.html`, `saved-listings.html`, `rejected.html` | Live Rightmove feed, fit-scoring, self-learning reactions. |
| **Intelligence** | `assets/js/refinement/`, `learned-preferences/`, `suggestions/` | Model Refinement Engine that learns the buyer's taste over time. |
| **Action** | `pages/outreach.html`, `journey.html`, `ask.html` | Outreach templates, the buying journey, and an AI "Ask" assistant. |

**Stack:** Zero-build static site — plain HTML + CSS + vanilla ES modules, all libraries via CDN.
Shared shell via fetch-injected partials (`components/`). Pico CSS v2 + design tokens. **Supabase**
is the live backend (auth + Postgres with RLS) for all user-state; a localStorage write-through
cache fronts it. Hosted on **GitHub Pages** (deploy on push to `main`). One Supabase Edge Function
(Deno/TypeScript) powers the "Ask" assistant.

**Current maturity:** Feature-complete through "v3" (live listings + Model Refinement Engine,
Stages 1–9). The codebase was modularised in a 2026-05 refactor: 132 JS modules, 51 CSS partials,
~65 test files. This overhaul builds on that foundation.

## 3. Safety & merge protocol (non-negotiable)

This is the contract that makes a top-to-bottom rebuild safe. Every sub-phase obeys it.

**3.1 One segment, one sub-phase at a time.** Never refactor two segments in one branch. Decompose
each segment into the smallest shippable sub-phases. A sub-phase that can't be described in a
sentence is too big — split it.

**3.2 Branch discipline.** This managed session develops on **`claude/fable-refactor-plan-yfmzmq`**.
For the actual rebuild program, follow `CLAUDE.md` §1: commit directly to `main` in small,
green-tested increments *unless* the session mandates a feature branch — in which case branch per
sub-phase, merge to `main` only when green, and never park long-lived divergent branches.

**3.3 Plan-mode contract (`CLAUDE.md` §14).** Before each sub-phase, enumerate in order: (1) files
to edit *and the sections within them*; (2) order of operations; (3) test impact (which tests change,
which are added, harness run green); (4) an explicit **out-of-scope** list. If scope changes
mid-flight — a new file is needed, a §16 guard-railed file is touched, a refactor surfaces — **stop,
surface the divergence, re-plan.** Do not power through.

**3.4 Guard rails (`CLAUDE.md` §16).** These files are never touched by feature work; modifying any
is its own named, separately-approved phase: `assets/css/tokens.css`, `assets/js/storage.js` +
`storage/*.js`, `assets/js/config.js`, `assets/js/data-loader.js`, `assets/js/finances.js` +
`finances/calc-*.js`, `assets/css/dashboard.css` (import shell), `data/schema/area.schema.json`,
`.github/workflows/*`. **Extend, do not rewrite** the storage and finance layers.

**3.5 Supabase sync (`CLAUDE.md` §18).** Any change to data, schema, or storage triggers the
MCP-first session-start freshness check and session-end push-and-verify ceremony. User-state writes
go through Supabase via MCP (never repo JSON). Schema DDL goes through `apply_migration` only.
The §6 sync tests fail a half-synced commit — never commit one.

**3.6 Test gate.** `node tools/run-intelligence-tests.mjs` is green **before every commit**. This
single command runs all intelligence tests + the Supabase sync suite. Extend tests *with* the code —
characterization tests first (lock current behaviour), then refactor under their protection.

**3.7 Verification without a browser.** There is no browser in the environment (`DESIGN.md` §4).
Verify by re-reading the diff and reasoning through layout/specificity/token resolution, then hand
the owner one short on-device check note for anything that needs eyes.

**3.8 Merge & report.** Merge to `main` only on green. Push with `git push -u origin <branch>`
(retry with backoff on network errors only). After each merge: tick §6, post a one-line progress
update. Do **not** open a PR unless the owner explicitly asks.

**3.9 Reversibility.** Prefer additive, behaviour-preserving moves. Keep each commit individually
revertible. Before deleting or overwriting anything you did not create, look at it first — if it
contradicts how it was described, surface that instead of proceeding.

## 5. Global conventions Fable must hold throughout

- **Design anchors (`DESIGN.md` §1).** Every view is *Stripe-docs* (editorial, content-led) or
  *Linear-dense* (data-rich UI) — never both, never neither. Name the anchor in every UI commit.
- **Tokens only.** No hard-coded hex, type px, or off-scale spacing in component CSS — everything via
  `tokens.css` custom properties (`--space-*`, `--text-*`, `--rec-*`, `--focus-ring`).
- **The five rules of the overhaul (`DESIGN.md` §5).** At-a-glance precedence; no isolated
  calculators; always-show-then-explain; no graphic without a verdict; visual cues replace text.
- **Responsive doctrine (`DESIGN.md` §6).** Mobile-first; `min-width` breakpoints 480/768/1024/1280;
  no transition in the iPad 600–800 band; `dvh`/`svh` not `vh`; container-queries for components; no
  inline styles; ≥44×44 targets. Mechanically enforced by `tools/lint-responsive.mjs`.
- **Accessibility floor (`CLAUDE.md` §11).** WCAG 2.2 AA: contrast, focus-visible never obscured,
  landmarks + skip-link, labelled forms, no colour-only signals, `<dialog>` for modals, reduced
  motion honoured.
- **Pico v2 conventions (`CLAUDE.md` §12).** Semantic HTML first; theme via `--pico-*` variable
  overrides, not source edits; dark mode via `data-theme` attribute.
- **Module layout (`CLAUDE.md` §19).** Small single-purpose modules; thin `page-*.js` coordinators;
  CSS mirrors the JS structure. Keep new code in this shape.
- **Content accuracy (`CLAUDE.md` §7).** Area/house content only after place- and type-specific web
  research, with `sources[]`. Never auto-generate. Only openly-licensed imagery.

### Data flows & shell mechanics

1. **Partial injection (`components.js` / `injectIncludes()`):** Every page declares `<div data-include="components/header.html">` etc. At page load, `components.js` fetches each partial as HTML, parses into template, replaces the div with template content. Race: all fetches concurrent; errors logged as comments. Success fires `shell:ready` custom event (used by page-* modules to coordinate startup).

2. **Navigation & active state (`components.js` / `setActiveNav()`):** All nav links carry `data-nav="pages/foo.html"` (relative to app root). `setActiveNav()` normalises the current location (strips index.html, trailing slashes), compares to resolved `data-nav` targets, sets `aria-current="page"` on match. CSS marks active via `::before` accent bar (left, 3px width).

3. **Theme cascade:** System preference is read via `matchMedia('(prefers-color-scheme: dark)')`. User override persists in `localStorage[${STORAGE_NS}:theme]` as 'light' or 'dark'. `effectiveTheme()` picks saved > system > 'light'. Theme applied via `document.documentElement.setAttribute('data-theme', 'dark')` or removed for system-default. Dark tokens map via `:root[data-theme="dark"]` selector and `@media (prefers-color-scheme: dark)` in `tokens.css` (both override same hues, same structure). Theme toggle button updates `aria-pressed` attr; SVG sun/moon icons swapped by CSS `:checked` / `[aria-pressed]` selector.

4. **Font loading:** Self-hosted woff2 subsets (unicode-range split), `font-display: swap` → swap text in fallback immediately. Three families: Fraunces (display, 400/600), Instrument Sans (body, 400/500), JetBrains Mono (data, 400/500). Loaded before base.css so no FOUT. `font-feature-settings` active on body + headings: "ss01" (Fraunces stylistic set 1), "cv11" (Instrument Sans contextual), "kern" (all), "liga" (headings); Fraunces adds "ss02" (serif alt).

5. **Auth guard redirect:** On every non-login, non-setup page, `auth-guard.js` imports `supabase-client.js` (throws pre-setup), checks session. No session → redirects to `pages/login.html?next=<current>`. Session exists on login page → redirects to `?next` param or home. Flash prevention: `<html data-auth-state="pending">` blocks body visibility (in index.html `<script>` before CSS); auth-guard removes attr once confirmed, revealing page synchronously (no flicker on auth'd pages, no DOM parse on unauth'd).

6. **Nav drawer (native `<dialog>`):** `initNavDrawer()` wires burger toggle + close button to `dialog.showModal()` / `dialog.close()`. Backdrop click closes. Any nav link click closes. Animations via `@keyframes nav-drawer-in` (translateX 220ms); reduced-motion honoured globally in `base.css`. Focus trapped by browser (native `<dialog>` feature).

7. **Header height dynamic var:** `initHeaderHeightVar()` reads `.site-header` computed height (default 64px), sets `--header-h` CSS var. Fires on resize (passive). Used by `:target` anchor-jump scroll-margin and focus-visible scroll-margin-top (§2.4.11 Focus Not Obscured compliance).

### Known smells / tech debt / risks

1. **Flash on pre-setup mode:** If `supabase-client.js` doesn't exist, `auth-guard.js` falls back silently (no session check). Page stays visible (data-auth-state removed). If the user then navigates to a page that needs Supabase (e.g., profile), it will fail to load data. No graceful error boundary exists; failure mode is silent empty state + console error only.

2. **Partial injection is synchronous-ish:** Fetches are concurrent but the DOM replacement is sequential and waits for all promises. If a partial is very slow or fails, the page is dark for a long time. No timeout; no fallback render (header/nav can't render inline, only via fetch). Accessibility risk: page isn't focusable until partials inject.

3. **Active nav path normalisation is fragile:** `normalisePath()` uses `new URL(p, location.origin).pathname` which may behave differently on GitHub Pages sub-paths vs localhost. The `data-nav` relative paths are resolved via `config.js` URL logic, but the comparison happens on normalised pathname. If the path parsing diverges, active state fails silently. No tests.

4. **Theme toggle label text mutation:** Button text is changed by JS (`btn.querySelector('.theme-toggle__label').textContent = ...`). If the markup changes (no class, or moved), the toggle breaks silently (aria-label is updated, but visible label stays stale).

5. **Hard-coded safe-area-inset values in base.css:** Header/nav/burger use `max(..., env(safe-area-inset-*))` for notch safety. If a new page-level container is added with custom padding, it must remember to apply the same inset logic or will overlap on landscape/notched devices. No shared CSS utility for this (Pico doesn't provide it).

6. **Responsive lint baseline is not enforced in CI:** The lint runs in `run-intelligence-tests.mjs`, but a new violation is only flagged if its count exceeds the baseline. A developer can add a second instance of an off-token value if one already exists (e.g., a second `12px` border-radius when the baseline has 1). The count-based approach masks this. The lint should be prelude/snippet-based, not count-based, to catch semantic regressions.

7. **Footer is static placeholder:** "Static prototype · your edits are saved in this browser." Footer text is hard-coded. No way to update it via data or config. No links, no versioning, no legal disclaimer placeholder.

8. **Header hamburger icon is custom SVG (not an icon library):** The hamburger and close buttons (in components/header.html and components/nav.html) are inline SVGs with hard-coded `stroke-width="1.75"`. If icon style guidelines change (e.g., weight → 2), both must be edited manually. No design-token tie-in. Close button has a different visual style (X instead of 3-line).

9. **No dark-mode SVG stroke handling:** The hamburger + close button icons use `stroke="currentColor"` (good), but other inline SVGs in the header (brand mark, sun/moon) also use currentColor but may not have been tested for legibility on both light/dark backgrounds. No documented test.

10. **Focus ring is static:** `--focus-ring` is a box-shadow (not an outline). Works for buttons/links, but some custom interactive elements (e.g., inline SVG with tabindex) may have invisible focus. No audit exists.

11. **View Transitions are opt-in per-page:** `@view-transition { navigation: auto; }` in `base.css` enables auto cross-document transitions. Named transitions (e.g., `area-title` morph) are applied by JS in page-areas.js via `.style.viewTransitionName`. If a page doesn't apply the name, the transition doesn't happen. No fallback; no warning. The allowed lint exception for `.style.viewTransitionName =` is permissive (count = 1, but multiple pages may set it).

12. **No error boundary for shell injection failures:** If a partial fetch fails (404, CORS, etc.), the page is left with a comment `<!-- include failed: ... -->` and continues. No warning to the user. The page may render broken (e.g., no header if header.html fails) with no indication why.

### Known smells / tech debt / risks

1. **13 unrelated tiles, one coordinator:** page-home.js blindly calls all 13 `render*()` functions. If one fails, no fallback per-tile. If one is slow, blocks others only if sync; async are parallel, but `renderAll()` is synchronous and calls all in sequence.
   - **Risk:** One tile error (e.g., `renderNba()` fetch fails) currently uncaught in `renderAll()`. Error does not prevent others.
   - **Smell:** Repetitive list of imports + calls; no registry pattern.

2. **Loading-state UX incomplete:** `markLoading()` dims 5 key headline elements (td-headline, tf-headline, etc.) before fetch; `clearStuckLoading()` restores to `—` if still empty after render. But:
   - Only 5 elements tracked (LOADING_IDS); other tiles (shortlist, criteria, journey, readiness) have no loading state at all.
   - **Risk:** If a tile fails silently, user sees stale data + no indication of error.

3. **`getFinances()` `onUpdate` callback:** Triggers `renderAll()` whenever Supabase cache refreshes. But:
   - No debounce — rapid refreshes (e.g., user typing in finances form on another tab) cause 13 tiles to re-render per keystroke.
   - **Risk:** Perf regression if storage layer chatters; tiles with SVG redraws (sparkline, fan, donut, ladder) are particularly expensive.

4. **Bento grid layout fragile:** Spans controlled entirely by media queries in tile-card.css (tile-deposit + tile-flow span 2 cols only at 1024+). But:
   - Arbitrary assignment — why only these two? Why not others?
   - Container-query threshold (720px) inside tile-deposit.css for arc left/right split **differs** from bento media breakpoint (1024px) — misaligned intent.
   - **Risk:** Future tile addition/removal or responsive refinement risks broken grid layouts.

5. **Mobile-first vs. container-query mismatch:** tile-card.css uses media queries for bento layout (1024/1280), but individual tiles (deposit, money-flow) use `@container (min-width: 720px)` for internal two-column splits.
   - A tile might be 720px wide (internal layout shifts) but bento is still single-column (1024px threshold not met).
   - **Smell:** Two independent responsive systems; hard to reason about breakpoint intent.

6. **SVG drawing in JS via string concatenation:** Every visual tile (affordability, money-flow, sparkline, scenarios-fan, donut, seasoning-bar) builds SVG strings with `svg +=` loops.
   - Inline `aria-label` + `data-*` attributes mixed in string builders (e.g., tile-affordability.js line 34–67).
   - **Risk:** Escaping bugs (addressed via `esc()` helper, but pattern is fragile); hard to refactor.

7. **Verdict badge colour coding in CSS:** `.verdict-badge--comfortable`, `.verdict-badge--stretch`, etc. are **tinted backgrounds + text colours in tile-extended.css**, not monochrome. But DESIGN.md §5 rule 5 says *"Existing accent / ink / paper palette only; never a seven-pastel palette."* The stretch band uses a custom orange hue.
   - **Smell:** Visual tokens leak into component CSS (tone should be single accent + paper mixes).

8. **Deposit scenario chips vs. affordability scenario tile:** Two separate concept renderings:
   - `tile-deposit.js` renders `.scenario-row` with 4 buttons (baseline/+200/+500/+5k) — **savings scenarios**, aria-pressed state toggled by JS.
   - `tile-affordability-scenarios.js` renders `.scenario-list` with 3 rows (buy sooner/target/stretch) — **affordability scenarios** with verdicts + ETAs.
   - **Smell:** Naming collision (both "scenarios"); conceptually different (savings vs. affordability); no clear distinction in UI.

9. **Readiness checklist priority list hardcoded in tile-readiness.js:** `READINESS_PRIORITY` is a 7-item array with exact keys + labels. If the schema changes (new readiness item, renamed key), this breaks silently.
   - **Risk:** Future Supabase schema change (new checklist item type) requires code edit + test update.

10. **ISA YTD in two places:** Stat rendered inline in deposit tile (line 71 of index.html) + as separate tile-isa-ytd.js module. Both fetch `getInvestments()`.
    - **Smell:** Duplication; if one needs refactoring, the other is forgotten.

11. **Affordability ladder hard-coded price range:** `LADDER_RANGE.min/max` (£100k–£2M) + `LADDER_RANGE.step` (£20k) baked in tile-affordability.js. If user's criteria exceed range, ladder clamps silently.
    - **Risk:** User sees ladder with no markers if their budget is £2.5M; no warning.

12. **No visual distinction between "loading" and "no data":** Both states render `—` (dash). If storage returns `null`, user can't tell if it's still loading or genuinely missing.

### Data flows

### Input pipeline
1. **Storage (Supabase via `storage.js`)**:
   - `getFinances()` → raw finances object (income, savings, bills, expenses, one-time costs, shopping list, gift cards, goal.targetDeposit, mortgage assumptions)
   - `getCriteria()` → criteria object (budget bounds, property type preferences, area filters)
   - `getInvestments()` → investments object (trading212ISA: currentPortfolioValue, earmarkPct, strategyEpochs, holdings)
   - `getGoals()` → goals object (timeline.horizon, deposit.hopedFor, area selections)

2. **Derivation (`finance-derive.js`)**:
   - Receives raw finances + optional investments
   - Computes 40+ derived fields: income aliases (takeHomeMonthly, totalMonthly, monthlyGross, bonusMonthly), line-item totals (ongoingBillsTotal.{monthly,annual}, expensesTotal.{monthly,annual,weekly}, oneTimeCostsTotal, etc.)
   - Cross-resource: `totalSavings = cashSavings + (isaEarmarkPct > 0 ? isaTotal × isaEarmarkPct/100 : isaTotal)` — gift cards tracked separately (not deposit-eligible)
   - Returns enriched object used by all downstream modules

### Chart rendering flows

**Savings-over-time** (`savings-series.js` + `section-v3-charts.js`):
- Input: `data/imports/trading212-history.json` (monthlySummary array: month/net/deposits/dividends/realisedPnL/interest, epochs)
- Shape: cumulative running sum of `net` per month, sorted YYYY-MM ascending, anchored to account-opened date
- Overlays engine baseline projection from `getSavingsVelocity()` (current value + monthly contribution rate projected forward)
- Chart.js line dataset: historicalMonths vs baselineProjection; target reference line at deposit goal
- Stub-safe: returns `{ isStub: true }` if history not imported

**Monthly deposits** (`investment-performance.js` → `section-v3-charts.js`):
- Input: same history JSON
- Bar chart: deposits (green) vs withdrawals (red) per month, capped at goal (£2,000/mo)
- Caption: "Net deposits per month — what you actually put in vs your £2,000 goal"

**ISA stacked-area** (`investment-performance.js` → `section-v3-charts.js`):
- Input: history.monthlySummary with running balances
- Four stacks: contributed (cumsum deposits − withdrawals), dividends, interest, market growth (currentValue − net − div − int)
- Time-series, smooth stacked area, percentage Y-axis optional

**Epoch comparison** (SVG, `investment-performance.js` output):
- Epochs from history.epochs (stockpicker: 2025-05-26→2026-01-01; etfCore: 2026-01-02→null)
- Per-epoch bar: width ∝ contribution, height ∝ return
- Interactive: hover for amounts

**Ticker treemap** (SVG, `investment-performance.js` output):
- Holdings data from history (if available)
- Squarified treemap: area ∝ capital deployed per ticker
- Colour by P&L (red/yellow/green)

**Realised vs unrealised P&L** (SVG):
- Two side-by-side bars: realised (locked gains) vs unrealised (market value − cost basis)
- Percentage gain label, breakdown in table

### Money-flow rendering

**Now flow** (`money-flow.js` + `section-flow.js`):
- Input: derived finData
- `getMoneyFlow(finData)` → { income.{takeHome, bonus, total}, buckets: [{name, amount, kind}], spare }
- Buckets: Bills, Expenses, Savings, Spare (always in that order per `FLOW_ORDER`)
- Rendering: SVG bar (300×40 viewBox, 4px pad, stacked rects) with labels > 36px wide; legend with swatches + amounts
- Spare shown as negative red bar if < 0

**After-move flow** (`section-later.js`):
- `getMoneyFlowPostMove(finData, monthlyMortgage)` — same shape, replaces Savings bucket with Mortgage
- Spare recalculated: income − bills − expenses − mortgage
- Side-by-side SVG bars: Today vs After Move, common maxTotal scale

### Affordability widget

**Input**: price (from slider 200k–600k, 2k step; or number input 100k–2m, 5k step)

**Computation** (`affordability.js`, `assessAffordability({ price, finances, criteria })`):
- Derive verdict from 3 independent signals, worst-band wins:
  1. **LTI** = loan ÷ gross annual income → band via LTI_BANDS (comfortable ≤4.5×, stretch ≤5.5×, tight ≤6.0×)
  2. **Payment%** = monthly P&I ÷ take-home % → band via PAYMENT_BANDS_PCT (≤40%, ≤52%, ≤60%)
  3. **Spare £** = income − bills − expenses − mortgage → band via SPARE_BANDS_GBP (≥£400, ≥£100, else tight)
- Secondary signals (not verdict drivers but flagged in whyVerdict): stressed rate (contract+3pp), LISA eligibility cliff (≤£450k), LTV tier (60/75/85/90/95)
- Output: `{ verdict, headline, loanRequired, ltvPct, ltvTier, depositGapToTier, monthlyPI, monthlyPIStressed, monthlySpareAfter, bandSignals, whyVerdict }` (15+ fields)

**Rendering** (`section-later.js`, `renderAffordWidget()`):
- Verdict pill colour: green (comfortable), yellow (stretch), orange (tight), red (out-of-reach)
- Grid: Required deposit, Loan needed, LTV (%) + tier, SDLT, LISA eligible, Monthly P&I, Stressed P&I, Spare after
- Why-verdict bullets (underlines, stress test flag, LISA cliff, spare warning, LTV gap opportunity)
- Affordability call-to-action: "Email these figures to a mortgage broker"

**What-if chart** (`getSavingsVelocity()` → `section-later.js`):
- Baseline + 9 scenarios (9 series in one Chart.js line chart)
- X: 0–60 months (or max ETA + 2)
- Y: savings balance (£)
- Legend: hoverable series, ghosted non-baseline
- Shows when each scenario hits the deposit target

### Known smells / tech debt / risks

### Architecture / design

1. **Siloed calculators anti-pattern (partially mitigated by P9 refactor)**
   - Pre-P9: calc-purchase, calc-lisa, calc-savings, calc-outlay lived as scattered utility functions in finances.js
   - Post-P9: split into `finances/calc-*.js` modules but finances.js is still a re-export shim (byte-identical surface)
   - Risk: If future refactors want to recombine or reorder these, they must maintain the shim signature or update every importer
   - Status: WATCH — shim is safe, but tight coupling via re-export means a rename of calc-purchase.js requires shim edit

2. **Affordability verdict engine is monolithic** (`affordability.js`, 320 LOC)
   - Single function `assessAffordability()` computes LTI + Payment% + Spare bands, derives deposit gaps, surfaces whyVerdict
   - No sub-module split (e.g., no affordability/verdict-lti.js, affordability/verdict-payment.js)
   - Risk: Hard to test or refactor a single band in isolation; future thresholds change requires careful re-read of 320 LOC
   - Status: ACCEPTABLE — the function is pure, well-commented, and the logic is linear (no mutual recursion). Test coverage is solid. Future refactors may extract band-logic helpers if complexity grows.

3. **Chart.js integration is sprawling** (`section-v3-charts.js`, 332 LOC)
   - 6 independent chart renderers (savings-over-time, monthly deposits, ISA stacked, div+int, epoch SVG, treemap SVG, realised P&L SVG) in one file
   - Risk: File is long; hard to find a specific chart renderer; risk of cross-contamination if shared helpers are added
   - Opportunity: Could split into `section-v3-charts/{savings-over-time, monthly-deposits, isa-stacked, dividend-interest, epoch, treemap, realised-unrealised}.js` — 7 files × ~50 LOC each. Would reduce read time + allow independent testing (if a test harness is built).

### Numeric precision & rounding

1. **2-decimal pound accuracy throughout**
   - `round2(n) = Math.round(n * 100) / 100` used in finance-derive, money-flow, savings-velocity
   - Risk: Large cumulative sums (e.g., 20-year mortgage projections) may drift by pence; bankers' rounding not used
   - Status: ACCEPTABLE for UX (display precision). If exactness is needed for regulatory output, a big-decimal library should be considered (future phase).

2. **LTV rounding at 1 decimal place** (`calcLTV()` in calc-purchase.js)
   - `Math.round((loan / propertyValue) * 1000) / 10` — matches lender conventions
   - Risk: 89.95% rounded to 90% may affect tier boundaries (e.g., gap-to-tier calculation)
   - Status: WATCH — verify rounding direction in affordability.test.js edge cases

3. **SDLT bands are integer-anchored** (e.g., £125,000 boundary is exact)
   - Risk: A £125,001 property triggers 2% on the entire band above, not just the slice; this is correct per GOV.UK but subtly different from a smooth function
   - Status: CORRECT — reflects actual SDLT rules

### Data flow & state management

1. **Investments optional in finance-derive** (line 59: `opts.investments || null`)
   - If investments not provided, `totalSavings = cashSavings` only (ISA earmark ignored)
   - Risk: If a page forgets to pass investments, deposit calculations will be wrong and give no warning
   - Status: WATCH — add a console.warn in finance-derive if investments is expected but null (future phase)

2. **Storage async but page-finances.js does not handle stale data**

   - Status: ACCEPTABLE for MVP. Future: wire `getCriteria()` and `getInvestments()` into the onUpdate pattern if needed.

   - Status: KNOWN — for UX purposes, the approximation is fine. Future: add a caveat in the UI ("approximate attribution").

### Data flows

### Fetch path (L1):
1. **Scheduled trigger:** `pg_cron` (noon Europe/London, via Supabase Vault token) or GitHub schedule backstop both dispatch `workflow_dispatch`.
2. **Manual trigger:** User taps "Pull listings" → `confirmFetch()` dialog → `requestListingsFetch(days)` RPC → GitHub Vault token → dispatch.
3. **Fetcher runs:** `tools/fetch-listings.mjs` — reads areas/*.json per-area files → extracts distinct outcodes → calls Apify actor (dhrumil~rightmove-scraper) → receives raw listings.
4. **Normalise:** `tools/listings-normalise.mjs` — Apify raw JSON → { rightmove_id, address, postcode, outcode, lat, lng, property_type, beds, baths, price, title, status, image_url, raw_json, added_date, price_history, … }.
5. **Validate:** `passesBaseline()` (classify.js) — type must be in allow-list (houses & bungalows broad), known price ∈ [250k–425k], known beds ≥ 2 (owner-set 2026-06-04).
6. **Dedupe & location:** `dedupeByRightmoveId()` + nearest-area match (haversineKm).
7. **Merge history:** existing listing's `price_history` + new price → upsert via PostgREST service-role UPSERT (rightmove_id conflict key).
8. **Result:** listings table = live source of truth; every row carries snapshot price history + raw_json details.

### Browse feed (L2):
1. **Load:** `getListings()` (storage.js) → pulls all rows from listings table, cached in localStorage.
2. **Score:** `scoreListingFit()` (fit.js) — affordability hard gate (out-of-reach → gated:true) + soft signals (beds, type, LISA, EPC, learned-prefs L4, rating).
   - Output: verdict ∈ {strong, possible, stretch, weak, reject}, score ∈ [0,1], gated bool, contributions[] (explainable).
3. **Partition:** `partitionFeed()` (feed-partition.js) — pure pipeline:
   - Radius filter (search-radius pre-check).
   - Hard gates (affordability + price floor, toggled by includeOOR).
   - Junk hide (auction / over-55, toggled by includeHidden).
   - Refinement hide (confirmed hide rules, toggled by includeHidden).
   - Decided suppression: `isDecided()` by rightmove_id + physical fingerprint (suppress re-lists).
   - Dedupe: `dedupeByFingerprint()` (fingerprint = type|beds|street|town, price-insensitive).
   - Apply controls (search/sort/filter).
   - Split reviewed (current reaction = like) vs unreviewed.
   - Return: scoredRows, visible (post-controls), unreviewed, reviewed, byVerb (like/pass/reject), counts.
4. **Render:** page-listings.js paints `visible` rows as cards: media, address, price, beds, type, geofence tags, flags (new-build/red-flags), fit-dot + verdict, why-expander, reactions, rating.
5. **Reaction:** User taps like/pass/reject → picker opens → optional reason chips (multi-select primary + sub-reasons) → Save → `saveListingReaction()` appends row to listing_reactions (append-only).

### Saved listings (L3a):
1. **Load:** `getReactionLog()` → all listing_reactions rows.
2. **Filter:** `latestPerListing(log)` → latest reaction per listing_id.
3. **Filter:** only `reaction = 'like'` rows → extract `listing_snapshot` (captured at reaction time).
4. **Dedupe:** `dedupeNewestByFingerprint()` (by snapshot address, newest-save first) — so a re-liked home under a new rightmove_id appears once.
5. **Render:** cards reuse .listing-card, surfaced like-reasons as read-only chips (great_area:quiet, good_value:underpriced, etc.), rating select, edit reaction/rating in-place.

### Rejected page (L3b):
1. **Load:** `getReactionLog()` → latest per listing.
2. **Filter:** `reaction ∈ {pass, reject}` → derive from `listing_snapshot`.
3. **Dedupe:** collapse same property (fingerprint) to most-recent decision.
4. **Render:** table (property type, area, title, price, reaction-verb, reasons, date-decided), search by type/area/address, 50-per-page paginate.

### Property dossier (L6):
1. **Load:** `getListing(rightmove_id)` → single row, expand raw_json.
2. **Extract:** `galleryImages()` (detail.js) → raw_json.images[] + image_url (deduped, primary guaranteed).
3. **Extract:** `floorplanImages()` (detail.js) → raw_json.floorplans|floorplan_url (if detail-page scrape).
4. **History:** `priceHistorySeries()` (detail.js) → time-ascending, per-step delta/pct/kind (listed/reduced/increased).
5. **Render:** Stripe-docs anchor — hero gallery (prev/next arrows, counter, thumb grid, full-screen lightbox modal), headline (verdict + fit-dot, price, title, place), Rightmove CTA + Google Maps, sections (only show if data: facts grid 2-col, EPC if known, council-tax if known, price history if >1 point, floor plan if available), full description prose, rail (fit-why open, reaction picker, status select, rating select).
6. **Reaction/status:** Same picker + picker-state (in-progress draft rehydrate), status select ∈ {new, saved, viewed, offered, rejected}.

### Learning loop (L4, seam reserved):
1. **Reactions feed learning:** `GRADED_REACTIONS = [like, reject]` → training signals.
2. **Training filters:** `isTraining()` (reaction-provenance.js) — exclude unattributed rejects, exclude removed_area + system reasons.
3. **Learned prefs:** `recomputeLearnedPreferences()` → derived preference weights per signal (area, type, etc.).
4. **Apply:** `listingLearnedPrefs()` → extract this listing's effective weights → `scoreListingFit({ learnedPrefs })` applies as soft contributions.

### Known smells / tech debt / risks

1. **Feed partition complexity:** `partitionFeed()` is a 100-line pure function with 10+ parameters (callbacks for scoring, hiding, etc.). It's correct but dense; a state-machine or builder pattern might clarify the gates.

2. **Fit-scoring opacity:** The "why" is built after the fact from contributions[]. A score of 0.65 lands in "possible", but there's no explicit mapping of score bands to *intent* — is 0.65 "good fit" or "edge case"? FIT_BANDS constants exist but may not reflect user intent.

3. **Reaction provenance:** The distinction between `reaction`, `reason` (scalar, reject-only, primary key), and `reasons` (array, for reject + like) is historically layered. Stored as dual-written (one-way compat path), but confusing to reason about.

4. **Dossier density (Stripe-docs anchor):** The facts grid, price history, floor plan, and full description are stacked vertically. On mobile, scrolling can feel long. Collapsible sections or a summary-detail pattern might help without breaking the anchor.

5. **Fingerprint false positives:** The fingerprint is deliberately conservative (need street + town). Edge cases: "The High Street, Fordingbridge" vs "Another High Street, Fordingbridge" will collide. Real but rare; acceptable trade-off.

6. **Price history UX:** If a listing has 20+ price moves, the history list grows. No UI truncation or chart view; purely list-based. Could become visually heavy.

7. **Learned preferences seam complexity:** L4 is not yet in the repo, but the scaffolding (learnedPrefs param, effectiveWeights lookup) is already wired into fit.js. The handoff point is clear but untested in the live flow.

8. **A11y of verdict dots & chips:** The fit-dot (colored circle) carries no text label on the card. Screen readers announce the verdict text, but the color alone (before text) is not sufficient for contrast/meaning check (§11). Tests exist but may not catch all edge cases.

9. **Gallery image loading:** No lazy-load strategy; all images (or at least a few) load eager on the hero. Responsive srcset missing; images not optimized for the aspect ratio (3/2 hero).

10. **Feed performance with many listings:** No virtual scrolling or lazy-render. A feed of 500+ unreviewed listings will cause DOM churn on every partition/sort/filter. Pagination or infinite-scroll not implemented.

11. **CSS cascade order-sensitivity:** listings.css imports 4 partials in cascade order (controls → cards → states → widgets). If a partial is moved or a new rule added out of order, specificity/cascade bugs are hard to spot. No Sass variables or computed order.

12. **Suppress.js usage:** Both `decidedSets()` + `isDecided()` are used, and `foldDecision()` mutates the sets in place. The three functions have different input/output shapes. Unifying to a single API (e.g., a class) might reduce confusion.

### Data Flows

### The Materialisation Pipeline (DB-Canonical)

**Core principle:** Supabase `areas` table is the source of truth (CLAUDE.md §18.5, relaxed owner decision). The repo materialised view lives at `data/areas/<id>.json` and is regenerated by a two-step pipeline:

1. **`sync-areas-from-supabase.mjs` (Step 0 — DB→repo)**
   - Fetches all `areas` rows (id + data jsonb) from Supabase via REST API (or from MCP-dumped file for dev).
   - Filters out household-onboarding stubs (source='household-onboarding', active=false) — runtime-only rows, never materialised.
   - Projects each row through `canonicalRecord()` to match build-areas' expected shape (defaults + nested field normalization).
   - Writes per-area files **only on semantic value change** (order-insensitive deep-equality check; nested key order from Postgres jsonb is ignored).
   - **Prune mode:** removes any per-area file whose id no longer exists in the DB (used after id/postcode migrations).
   - Writes parity snapshot `data/snapshots/areas.json` — compact rows {id, name, postcode, coords, coordsSource, active} for offline regression testing (tests/areas-db-repo-parity.test.js).

2. **`build-areas.mjs` (Step 1 — repos→files)**
   - Reads source CSVs: `data/source/villages.csv` (id generation, 192 base entries) + `postcode-regions.csv` (hubCity, regionDir, fallback coords).
   - Merges with **prior** per-area detail files (preserves all enriched content: overview, character, amenities, schools, transport, prices, images, sources, etc.).
   - **Price-summary baking:** extracts avgDetached/avgSemi/avgTerraced/avgFlat into a lightweight `priceSummary` sub-object shipped in the index so areas.html can compute fit verdicts without loading every per-area file.
   - Generates `data/areas.json` (INDEX_FIELDS only — ~30 KB, ships to client).
   - Generates per-area `data/areas/<id>.json` (DETAIL_FIELDS — up to 6KB each, lazy-loaded on detail page).
   - Regenerates `docs/AREAS.md` (human-readable county→town breakdown; auto-generated, not hand-editable).

**After DB write via MCP (e.g., area property edit, id/postcode migration):**
```
mcp execute_sql → update areas set data = ... where id = 'X'
  → node tools/sync-areas-from-supabase.mjs --prune
  → node tools/build-areas.mjs
  → node tools/verify-area-coords.mjs --online
  → node tools/run-intelligence-tests.mjs
  → commit + push
```

### Matching & Enrichment (Household-Added Areas)

**Flow (Phase 2 onboarding):** User picks a village via place-lookup → `matchCatalogArea()` decides link-or-stub → `enrichPatch()` fills coords/postcode/town/geofence → stub created in Supabase user-state.

- **`matchCatalogArea(place, catalog, maxKm=1.5)`:** Tries to find the exact place in the curated catalog.
  - Name match (normalized) required.
  - County match (normalized); if both known and disagree → no match.
  - Distance <= 1.5 km (haversine) OR postcode-district match → candidate.
  - Closest qualifying candidate wins (prevent like-named village cross-link).
  - If no match: stub created, no pre-linked catalog area.

- **`enrichPatch(candidate, pcRecord)`:** Builds the additive field patch for a stub.
  - Coords: prefer candidate lat/lng; fall back to postcodes.io postcode record.
  - Postcode: from candidate or pcRecord.
  - County: from candidate; fall back to pcRecord admin_county/region.
  - **County mismatch flag:** if candidate county + pcRecord admin_county clearly disagree (neither substring), set coordsSource='postcodes-io:county-mismatch' (soft block; user can confirm to unblock).
  - Town: from pcRecord admin_district (better than county-as-town).
  - geofenceRadiusMi, searchRadiusMi: hardcoded 3 (resolved-areas default).
  - coordsSource: 'postcodes-io:places+reverse' (or 'postcodes-io:postcode' if postcode-only lookup).
  - Soft-fail: if pcRecord is null (postcodes.io unreachable), return coords-only + coordsSource='postcodes-io-provisional' (stub is NOT yet fetch-eligible until enriched again later).

- **`isFetchEligible(area)`:** Predicate shared by dashboard classifier + fetcher.
  - Has coords (lat/lng non-null).
  - Postcode can derive outcode (via `postcodeDistrict()`).
  - No county-mismatch flag in coordsSource.

### How the engine works (end to end)

**Layer 1 (audit):** Append-only `listing_reactions` table. Every interaction (like/pass/reject) is one row, with an optional `listing_snapshot` (frozen listing state at reaction time) and `reasons` (feature-specific dismissals for rejects). Administrative reactions (area removed, price filtered) are marked with `reason='removed_area'` etc. and excluded from training.

**Layer 2 (derive):** 
1. **Aggregation (§2.1–2.2):** `buildAggregates()` reads reactions; extracts dimension values (area_id, property_type) from snapshot or live listing; applies exponential time-decay (half-life 150 days) to each reaction; computes effective counts (n_eff, k_eff) per value. Rejects only; passes ignored by design.
2. **Scoring (§2.3–2.8):** `scoreFromAggregates()` computes per-value metrics: point estimate p_hat, Wilson lower bound (95% confidence, continuity-corrected below n_eff=30), lift (p_hat / baseline reject rate), two-proportion z-test p-value, Benjamini-Hochberg FDR per dimension. Five deterministic gates:
   - **Gate 1 (global):** system_decayed ≥ GLOBAL_MIN_FEEDBACK (300) + dim_decayed ≥ DIM_MIN_FEEDBACK (150).
   - **Gate 2 (sample):** n_eff ≥ MIN_EFFECTIVE_SAMPLE (12) + distinct_rejected_listings ≥ MIN_DISTINCT (6).
   - **Gate 3 (confidence):** wilson_lower ≥ WILSON_FLOOR (preset: 0.72–0.88).
   - **Gate 4 (disproportionality):** FDR-significant + lift ≥ MIN_LIFT (preset: 1.15–1.6).
   - **Gate 5 (persistence):** consecutive runs qualified ≥ PERSISTENCE_RUNS (preset: 2–5).
3. **Ranking (§2.8):** wilson_lower desc, lift desc, n_eff desc, value asc. Volume-artefact flag when k_raw ≥ 30 AND lift ≤ 1.0 (high sample but baseline reject rate—not disproportionate).

**Layer 3 (persist):** `persistence.js` reads existing `refinement_suggestions` rows; maps prior runs_qualified → fed to engine → advanced by 1 if pass this run. Status resolved: engine controls 'actionable' (pass all gates) vs 'forming' (confident but not yet persistent); user-owned statuses ('confirmed_hide', 'confirmed_scrape', 'dismissed', 'snoozed') are STICKY (engine never re-raises). Emits idempotent SQL UPSERT (on CONFLICT guards user-owned statuses).

**Layer 4 (learn—refined preferences):**
- **Signals:** `signalsForListing()` extracts type/beds/baths/outcode/area_id/price_band/outdoor/parking. Feature inference for outdoor_space + parking from description text (conservative: abstain when ambiguous).
- **Weights (Layer 2 derive):** For each signal, count likes vs total graded reactions; apply same time-decay + confidence shrinkage (SMOOTHING denominator 3). Unattributed rejects (no reason) do not train—they hide the listing but don't move weights. Attributed rejects contribute at UNATTRIBUTED_DISCOUNT (0.35×) to non-implicated signals.
- **Weights (Layer 2⊕3 effective):** `effectiveWeights()` merges static FIT_WEIGHTS (affordability, beds, type, price, LISA, EPC, rating) + learned weights. Learned weights capped at MAX_LEARNED_WEIGHT (0.30); cold-start (gradedCount < 10) falls back to static fit + diversification.
- **Training progress:** `trainingProgress()` is balance-aware: like/reject counts, balance factor (min share / 0.5), milestone tracking (warming-up → learning → usable → solid → mature). Imbalanced flag (likeShare < 0.2) triggers "add more likes" next-action.

**Layer 5 (recommend—meta-observations):**
- **Conflict detection:** `detectConflicts()` surfaces when likes contradict stated criteria: (1) over-budget, (2) excluded type, (3) below-min-beds, (4) geofence tighten, (5) stop searching. 3-condition trigger: ≥MIN_CONFLICT_LIKES (3) likes, ≥MIN_CONFLICT_SHARE (0.6) of comparable likes, ≥1 within CONFLICT_RECENCY_DAYS (30 days). Dismissed prompts stay quiet for 14 days.
- **Next-best-actions:** `computeNextBestActions()` returns ordered list (cold start → strong un-reviewed matches → saved-unviewed → recent to review). Pure, inject scoreOf() closure for listing-fit scoring.

**Stage 3 (schedule):** `tools/refinement-run.mjs` reads reactions (file mode: JSON bundle from MCP execute_sql; REST mode: PostgREST), filters to genuine reactions (drops bulk area/price sweeps + admin removals via `reaction-provenance.js`), runs engine, plans persistence, emits SQL (never executes).

**Stage 4 (UI—read):** `page-refinement.js` reads engine suggestions + run meta + probation rows; renders inbox (max 5), patterns-forming, active (hidden), probation (paused), dismissed, snoozed. Model-confidence meter tracks training progress toward 300 decayed reactions.

**Stage 5 (UI—write: display-hide):** "Hide these from view" → `hideSuggestion()` writes a rule to `learned_preferences.overrides.__refinement_hidden`; status flips to 'confirmed_hide'; engine never re-raises. Restore via `unhideSuggestion()`.

**Stage 6 (UI—write: scrape-probation):** "Stop searching this area" → `stopSearchingArea()` writes a `scrape_probation` row (status='confirmed_scrape'); `tools/fetch-listings.mjs` subtracts probationed areas from the active scrape set. Re-probe cadence (every 6 scraper runs by default) temporarily re-includes area for exploration; if rejects stay high, status flips to 'reconsider' (user can bring back or leave paused).

**Stage 7 (learner control):** Sensitivity preset (Cautious/Balanced/Aggressive) persisted in `learned_preferences.overrides.__refinement_settings`; survives retraining; chosen by user via radio buttons on Refinement page.

### Known smells / tech debt / risks

1. **Opacity of the model:** The user sees "Rejected 42% — 15× your usual reject rate" but does not see which signals (beds, price, outcode) moved the weight. A listed home's contribution to a signal's weight is invisible. This asymmetry can make the model feel magical or untrustworthy.

2. **Weight drift over time:** Learned weights are derived fresh on every run (no persistence of per-signal weight state); they only live in `learned_preferences.weights` (in-memory during the page session). If the user's taste shifts (e.g. abandons a price band entirely), old signals can linger with stale weights until overwritten. The HALF_LIFE_DAYS (30) mitigates this but is fixed, not adaptive.

3. **Cold-start UX:** With <10 graded reactions, the engine doesn't fire and learned weights are ignored. The feed stays static. A new household has no signal until they react to ~10 homes, which is a long time with no feedback loop. The "Still learning" meter is honest but feels slow.

4. **Unattributed rejects poison the model:** If a user quick-rejects 50 homes without reason, those 50 do not train—they're filtered out. But a user might reject for a genuine reason they didn't articulate (e.g. "too remote" but never selected that reason). An unattributed-reject is binary: either it trains or it doesn't. No middle ground.

5. **Statistical testability:** The engine runs once per household per CI schedule (unclear: daily? weekly?). There's no hook to run it ad-hoc during development. The bundled test data in fixtures is synthetic; a real-world run is hard to replicate locally.

6. **Probation enforcement is async:** A user stops searching an area (writes a probation row). The *next* scraper run (tools/fetch-listings.mjs) will respect it. Until then, the area is still scraped. The delay is silent—the feed doesn't visibly react.

7. **Meta-observations rely on snapshots:** Conflict detection works off `listing_snapshot` (the frozen listing state at reaction time). If a listing was liked 90 days ago and the user's actual budget has changed, the snapshot is stale. The conflict won't surface until a *new* like at the new price triggers evaluation.

8. **Scope invariant is passive:** `refinement-scope-check.mjs` only *checks* invariants; it doesn't fix them. If a probation area is reactivated in `areas.json` without unpausing it, the invariant fails (probationedButActive is non-empty). The fix is manual.

9. **Learned weights can contradict fitted:**  A listing might score "strong" on static fit (affordable, correct bed count) but "weak" on learned fit (the user hates that property type). The UI shows the combined score, but the tension is not transparent. A user might wonder "why was this ranked low when it matches my criteria?"

10. **Testability of the "real" engine:** The unit tests mock reactions; they don't test against live listing data (no access to Rightmove API in test harness). Cold-start scenarios, edge cases with real price distributions, are not covered by the test suite.

4. **Unattributed-reject soft signal (P10d):** Instead of filtering unattributed rejects entirely, treat them as 0.35× weight (like non-implicated signals in attributed rejects). This gives them *some* causal signal while discounting wild guesses. Requires careful threshold tuning to avoid noise.

5. **Conflict snapshot freshness (P10e):** Before surfacing a conflict, check if the home's snapshot price has drifted from the current live listing. If a liked home's price dropped, lower the conflict priority or surface a "budget ceiling may have fallen" observation instead. Requires a fast price-lookup (cached, not full re-scrape).

6. **On-demand engine run (P10f):** Wire `tools/refinement-run.mjs` to a UI button "Analyze my reactions now" (on the Refinement page). User clicks, bundle is built via MCP, SQL is emitted, suggestions update in-place. Enables faster iteration for testing. Requires MCP integration + refresh hook.

7. **Per-signal snooze (P10g):** Instead of snoozing the whole suggestion ("Stop searching Brighton for 30 days"), allow snoozing *per signal kind* ("I like Brighton but hide detached homes there for 30 days"). Requires storing snooze rules per dimension:value:signal-kind and evaluating them on fit-score time.

8. **Probation re-probe UX (P10h):** When a probationed area reaches 'reconsider' status, the Refinement page shows "Latest batch suggests this might be worth another look." User taps "Re-enable" → immediately un-pauses + triggers a re-probe. Currently re-probes are silent; surfacing them closes the feedback loop.

9. **Learned-weight persistence for analysis (P10i):** Snapshot `learned_preferences.weights` at the end of each run (store in a `refinement_runs.weights_snapshot` jsonb column). Enables post-hoc analysis of weight evolution, debugging of drift, and charting "your taste over time". Requires adding a column + updating the plan.

10. **Sensitivity preset A/B testing (P10j):** Log which preset the user chose + which suggestions they acted on. Analyze if Aggressive users convert more / faster than Cautious users, or if they're just noisier. Informs calibration of the preset matrix. Requires adding a `user_preset` column to refinement_suggestions.

### How it works (end to end)

1. **Browser (frontend):** User types a question on pages/ask.html. page-ask.js (coordinator) collects the text turn and prior user/assistant turns (text only, no tool blocks) into the message thread.

2. **ask/client.js (transport):** Fetches `POST https://qxmyrahqsopmaeokxdub.supabase.co/functions/v1/ask` with:
   - Authorization header: Supabase JWT from supabase-client.js
   - Body: `{ messages: [{role:'user'|'assistant', content:'text'}], model?: 'claude-haiku-4-5'|'claude-sonnet-4-6' }`
   - Returns SSE stream; parses into async iterator yielding `{type:'text'|'tool'|'done'|'error', …}` events.
   - AbortController stops mid-stream.

3. **Edge Function (index.ts):**
   - Verifies JWT; resolves household_id from household_members RLS-scoped query.
   - Calls `buildSystemPrompt(supabase, householdId)` → static cached block + dynamic context block.
   - Calls Anthropic `/v1/messages` with:
     - Model: default `claude-haiku-4-5` (cheapest), fallback from `claude-sonnet-4-6` if requested; Opus not allowed.
     - max_tokens: 1024 (runaway backstop).
     - system: prompt blocks (first marked cache_control:ephemeral for ~90% cost savings).
     - tools: TOOLS array from tools.ts (13 read-only tool definitions, 4 with strict:true + additionalProperties:false).
     - messages: sanitized + capped at MAX_HISTORY_TURNS (24), per-turn char cap 16k, ensure starts with 'user' role.
   - Relays Anthropic SSE → client SSE (text deltas as {type:'text'}, tool calls as {type:'tool',name}).

4. **Tool loop (index.ts + tools.ts):** Runs ≤ MAX_TOOL_LOOPS (6) times:
   - If Anthropic stop_reason is not 'tool_use', emit {type:'done',usage} and exit.
   - Otherwise, accumulate tool_use blocks, call `runTool(name, input, ctx)` for each (synchronous, ≤6 calls per turn).
   - pure.js handlers (filter/rank/search/shape) run on fetched data; tools.ts wraps RLS-scoped Supabase queries.
   - Return tool_result blocks to Anthropic; loop.

5. **Streaming + replay (ask/transcript.js):**
   - beginAssistant() creates a live bubble with aria-live polite.
   - token(text) → appends to .innerHTML (mdToSafeHtml, escape-first).
   - tool(name) → updates status line "Checking {label}…".
   - end() → finalizes text + appends "Sources: …" line listing tools used.
   - error(msg) → shows alert and ends.

6. **Persistence (page-ask.js → storage/ask.js → Supabase):**
   - After stream ends (done/error), if the assistant turn has content or tools, page-ask.js calls:
     - createAskConversation(title, messages) if new, or saveAskConversation(id, {messages}) if existing.
   - storage/ask.js sends UPSERT via RLS-scoped Supabase client (household_id filter).
   - Messages persist as `[{role:'user'|'assistant', content:'text', tools:[...]}]` (tool blocks reconstructed at display time from the stored `tools` array, not from Anthropic's intermediate blocks).

7. **History (history.js):**
   - listAskConversations() → list by household_id, ordered by updated_at desc, show title + timestamp.
   - Switch/rename/delete actions all go through storage/ask.js.
   - Clicking a thread calls onSwitch(id) → page-ask.js loads via getAskConversation + replays bubbles.

### Known smells / tech debt / risks

### Prompt brittleness
- **System prompt is prose.** The STATIC_PROMPT in prompt.ts is a long English block with no schema enforcement. A refactor to structured prompt format (e.g. JSON with separate identity/domain/vocabulary/safety sections) would make it easier to version, test, and diff. Currently, any change must be hand-audited for ambiguity.
- **UK FTB facts hardcoded.** LISA cap (£450k), Stamp Duty rules (April 2025), survey costs — sourced from docs/CONTEXT.md but copied into the prompt. A single source of truth (e.g. a data/static/fib-facts.json + a prompt builder) would reduce drift risk.
- **Vocabulary drift.** TOOL_LABELS in transcript.js must match tool names in tools.ts; listing verdicts (strong/possible/stretch/weak) must match FIT_BANDS and bandForScore logic in pure.js. No schema enforcement — a refactoring pass could enum these.

### Model / version config
- **Model string hardcoded in index.ts.** DEFAULT_MODEL = "claude-haiku-4-5", ALLOWED_MODELS = Set(["claude-haiku-4-5", "claude-sonnet-4-6"]). If Anthropic releases claude-fable-5 or a new-tier model, the allow-list must be updated and re-deployed. No feature flag / environment-driven config. Consider a Supabase config table (org_settings: model_allow_list, default_model) so a non-code deploy can pin a new model.
- **No thinking parameter sent.** The prompt explicitly says "No thinking parameter is sent, so thinking stays off (zero thinking tokens)." If a future model (e.g., claude-opus-5) has default-on thinking and the contract changes, the max_tokens ceiling (1024) will be consumed by thinking first, starving actual tokens. Safe for now; monitor at redeploy.
- **claude-opus-4-8 explicitly removed.** ASK.md §5 says Opus's default-on thinking is the wrong shape. This is a user-intent constraint, not a technical block. Document why in index.ts (currently just an omission from ALLOWED_MODELS).

### Tool safety / scoping
- **Read-only tools.** All 13 tools are read-only; draft_outreach returns text, never sends. BUT:
  - Prompt-injection via listing descriptions / area notes is treated as data, not instructions (mdToSafeHtml escape-first, tool data marked as data in prompt). This is correct but fragile — depends on the model respecting the "treat as data" instruction. A refactoring pass could add a `@noexec` or `sanitized:true` marker to each tool's output so injection intent is harder to exploit.
  - query_listings now push-down indexed predicates (status, price range, area substring) to Postgres (plan.filters) before pure.js ranks. The filter builder (sanitizeFilterTerm) strips commas, parens, % — but is this exhaustive against PostgREST .or() / .eq() / .ilike() syntax? A second pair of eyes (code review) recommended before claiming it's hardened.
- **Tool argument schema.** 4 tools (query_listings, get_listing, search_areas, get_area) use strict: true + additionalProperties: false. draft_outreach is non-strict (listing/contact/extra free-form by design). Current Anthropic API version (2023-06-01) supports strict tool use; ASK.md §5 says to "smoke-test after deploy to confirm the current API line needs no beta header for `strict`." Not currently validated in CI.

### Streaming & latency
- **SSE stream never closes on tool_use.** Correct design (the client doesn't see the tool call as "done" mid-answer), but:
  - If a tool call fails and the model stops (e.g., an RPC error), the client receives {type:'error'} and the stream closes. Conversely, if the stream is interrupted mid-delta, the client shows "interrupted" error (in ask/client.js). Recovery is a fresh "Re-run the question" (user retry), not resumption. This is acceptable for a read-only tool set.
  - Tool execution (runTool) is synchronous in the loop; if a tool hangs, it blocks all downstream tool calls. Max timeout is the Edge Function's hard timeout (varies by Supabase plan, typically 10–60 s). Consider async timeouts or a tool-call budget (time, not just count).
- **Prompt-caching benefit depends on question distribution.** The static block is marked cache_control:ephemeral, so repeat questions within ~5 min reuse the cache for ~90% savings. If a user asks only once per day, cache hit-rate is 0. The dynamic context block is always recomputed (it's small, ~200 tokens). This is efficient; no risk, but monitor cache_read vs cache_write usage logs (index.ts logs cache_read / cache_write / thinking).

### Error handling
- **Error surfaces as {type:'error', message}** in SSE stream, rendered as a red alert in the transcript. No retry logic at the client level — user must click "New chat" or re-send the question. For a read-only tool set (no state corruption risk), this is acceptable.
- **Tool failures return {error: "..."}** and the model receives the error as a tool_result. The model may hallucinate a recovery or apologize. No explicit tool-failure budget or escalation (e.g., "if 3 tools fail, stop"). The model generally handles "tool X failed" gracefully, but hasn't been extensively tested.
- **Anthropic API errors (non-200 status)** are logged in index.ts and relayed as {type:'error', message, detail}. The detail is truncated to 500 chars to prevent token waste. No alerting or automatic retry.

### Deno Edge Function deploy path
- **No local dev environment.** The Edge Function code (index.ts, tools.ts, prompt.ts, pure.js) runs only in the Deno runtime on Supabase; it's not tested locally (no local Deno REPL wired into the harness). A refactor to split pure.js (already portable) from tools.ts (Deno-specific) would allow offline unit tests of the query builders + tool logic without a local Deno runtime, but the RLS-scoped Supabase integration still needs an online test.
- **Deploy via supabase functions deploy ask.** The source is version-controlled (supabase/functions/ask/); the dashboard editor is ignored. A stale deploy (developer runs tests locally, edge code drifts, a git pull misses the deploy step) is a known failure mode. No CI/CD hook (e.g. "deploy on push to main"). Deploy is manual + documented in ASK.md §4. Consider adding a GitHub Action or a pre-commit hook reminder.
- **ANTHROPIC_API_KEY is a Supabase secret.** Stored securely, never in the repo. A leaked key can be rotated via the Anthropic Console. The Supabase secret must be set once at deploy time (supabase secrets set) and is then baked into the deployment. No per-turn key rotation.

### Security of tool outputs
- **No PII masking.** Tools return real listing descriptions, area overviews, profile first names, etc. These are the user's own data (RLS-scoped to their household), so leaking to Anthropic is acceptable. BUT:
  - Streaming markdown via mdToSafeHtml is escape-first (safe against injection).
  - The Anthropic API call is made from the Edge Function (never client-side), so the API key is not exposed.
  - Tool data is marked "data, never instructions" in the prompt, so the model is (theoretically) less likely to follow embedded commands. This is a soft control, not a hard one.
- **No audit log.** Each Ask call logs `usage: {input, output, cache_read, cache_write, thinking}` to Deno logs (visible in Supabase logs). The full conversation is persisted to ask_conversations, but there's no explicit audit of which tools were called, what data they returned, or whether an answer was accurate. Manual review of usage logs + conversation history is possible but not automated.

### Summary

The **Ask assistant** is a stateless, tool-driven conversational layer over household Supabase data, powered by Claude Haiku 4.5 via a Supabase Edge Function. The entire filtering/ranking logic lives in pure.js (unit-tested, portable), making it cheap (Haiku tier) and deterministic. The backend is tightly scoped (13 read-only tools, RLS-guarded), the frontend is modular (composer/transcript/history/client as separate concerns), and persistence is simple (RLS-scoped Supabase writes via storage/ask.js). Main refactor opportunities are: model upgrade (consider Fable 5), structured prompt, config table for non-code model switching, async tool timeouts, and integration tests. Known brittleness: prose prompt, hardcoded model strings, and no client-side tests. Security is sound (escape-first HTML, data-not-instructions marking, no PII on edge), but audit logging is minimal.

### Coupling & dependencies

1. **profile-schema.js is the canonical contract**
   - All profile readers + writers flow through `canonicalProfile()` (enforces nested shape) or `normalizeProfile()` (adds flat outreach mirrors).
   - Tolerant of three historical shapes (flat-full, nested, summary-only); never loses data.
   - Display helpers derive human prose from structured data if no freeform summary exists.
   - Idempotent: `canonical(canonical(x)) === canonical(x)`.

2. **Criteria schema shared across finances/listings/areas**
   - criteria.budget (max/min/offerTarget/offerStrategy) drives affordability filters in listings.
   - criteria.location.areaRadiusOverrides override the per-area radius from content `areas` table (via setAreaRadiusOverride + clearAreaRadiusOverride).
   - criteria.propertyTypePrefs (preferred/acceptable/excluded) filter listings + areas.
   - criteria.mustHaves/niceToHaves appear in outreach templates.

3. **Journey checklist deep-links into outreach**
   - tasks.outreachTemplateId points to a template id (e.g., A1, B2).
   - Click on step row → if task has outreachTemplateId, offer "→ Email" link to `outreach.html?templateId=<id>`.
   - Outreach page deep-link handler opens the template dialog immediately.

4. **Setup wizard state machine → profile branching**
   - Wizard steps depend on profile.buyingSituation (first-time-buyer / home-mover / cash-buyer) and employment.basis.
   - Cash buyer: skips income + mortgage steps.
   - Self-employed: reveals selfEmployment.structure + yearsActive fields (hides employed-only fields like employer).
   - Joint (household.applicants === 2): reveals applicant2 fields.
   - Each step's visibleFields() returns a filtered list; requiredGate checks only visible fields.

5. **storage.js re-export shim (§16 guard-rail)**
   - storage.js is a byte-identical shim over storage/{core,user-state,listings,outreach,refinement,ask}.js.
   - Per §16, storage.js is never rewritten; only the internal split modules are extended.
   - All profile/criteria/finances/goals/journey/contacts/outreach reads + writes go through storage.js accessors.

6. **Outreach template resolution depends on profile-schema normalization**
   - Templates declare `dataNeeded` as flat paths (profile.firstName, criteria.mustHaves, finances.aipAmount, etc).
   - outreach-renderer.js resolves these from `normalizeProfile()` output (which adds firstName/lastName/mobile/email/postcode mirrors).
   - If a path is missing, the template shows a `[missing data]` placeholder.
   - Quantity-of-Information Ladder: each recipient role gets only the paths in their template's `dataNeeded`.

### Known smells / tech debt / risks

### 1. Three page-profile-*.js files (overdue split?)
- `page-profile.js` ≈ 260 lines: About you card (read + edit dialog).
- `page-profile-detail.js` ≈ TBD: Application detail card.
- `page-profile-page.js` ≈ TBD: Aggregate controller.
- Risk: Three entry points, unclear responsibility boundaries. Candidate for merge under §19 module layout (one page-profile.js coordinator + profile/ submodules for detail/about/application).

### 2. Wizard complexity (1688 LOC across 6 modules)
- Steps are declarative (good), but branching via `visibleFields()` + `includedSteps()` is implicit.
- A field like `profile.selfEmployment.structure` is hidden for employed, shown for self-employed, but the rule lives in steps.js conditionals.
- Risk: Hard to audit what a given profile state includes; no visual "what if I change to self-employed?" preview.
- Opportunity: Swap out steps.js for a simpler reactive model (e.g., expose visibleFields as a computed property, allow the UI to query "does this field appear for state X?").

### 3. Template maintainability (30+ templates + 5KB JSON)
- Each template has id, stage, role, subject, body, dataNeeded, tone, bestPracticeNotes, sources (all live in one big JSON array).
- dataNeeded is manually written per template; no automated check that body placeholders match dataNeeded keys.
- Risk: A template body references {{profile.middleName}}, but dataNeeded doesn't list it → renderer shows blank.
- Opportunity: Generate dataNeeded by parsing body template (find all `{{...}}` keys), then validate.

### 4. Journey data staleness
- `data/journey.json` is static repo JSON, not a Supabase table.
- If the real buying journey changes (e.g., a new step needed between MIP and offer), the JSON is hand-edited.
- Risk: Outdated guidance if content lags behind user experience.
- Opportunity: Move to Supabase content table (like areas/house_types), allow owner to edit via the portal or MCP.

### 5. Outreach form a11y (field labels + screen-reader announcements)
- Dialog has context fields (contact, property, viewing-date) — unclear if all have programmatic labels.
- Preview pane updates on input change; no ARIA live region to announce "preview updated".
- Risk: Screen-reader users may miss that placeholder substitution happened.
- Opportunity: Add `aria-live="polite"` to preview, announce "email refreshed with new contact details" on context change.

### 6. Profile-schema forward compatibility (unknown future fields)
- The CONSUMED set guards against data loss: any key not in CONSUMED is passed through verbatim.
- Risk: If a future field is added but not listed in CONSUMED, it gets absorbed into an object (e.g., misplaced under person instead of top-level).
- Opportunity: Test this exhaustively; consider a strict mode that warns if an unknown key is seen.

### 7. Criteria save-bar (inline edit mode) vs dialog (profile edit)
- Profile uses a modal dialog for edits.
- Criteria uses a sticky save bar + inline edit mode.
- Different UX paradigms for the same data class.
- Risk: User confusion; inconsistent save affordance.
- Opportunity: Converge on one: either profile to save-bar, or criteria to dialog.

### 8. Outreach contact picker UI (not yet examined in detail)
- The dialog has a `<select>` with a "select or type" affordance.
- Risk: Unclear if the UI is accessible, if typing creates a new contact inline, or if it requires a separate add-contact flow.
- Opportunity: Verify a11y; consider splitting contact picker from template dialog if it's complex.

### Architecture & data flows

### Four-class data model (CLAUDE.md §18.1)

1. **User state** (21 tables, per `household_id`): source of truth = **Supabase**. Portal writes via `storage.js` → localStorage write-through → `_sbUpsert`; Claude via MCP `execute_sql`. Never in repo JSON. Tables: `profile`, `criteria`, `finances`, `goals`, `shortlist`, `zones`, `journey_checks`, `journey_progress`, `contacts`, `outreach`, `readiness_checklist`, `investments_accounts`, `investments_history`, `debts_credit_cards`, `debts_student_loans`, `debts_other`, `listing_reactions` (append-only), `learned_preferences` (recomputed), `area_confirmations` (blob), `household_areas` (relational), `ask_conversations` (chat threads).

2. **Content — areas**: source of truth = **Supabase `areas` table** (DB-canonical, §18.5 relaxation 2026-06-04). `data/areas/<id>.json` is a **materialised view** regenerated by `tools/sync-areas-from-supabase.mjs` after every DB write. Parity guarded by `tests/areas-db-repo-parity.test.js`.

3. **Content — other** (`house_types`, `checklists`, `outreach_templates`): source of truth = **repo JSON**. Only `house_types` has a Supabase mirror; `checklists` and `outreach-templates` are repo-only. `data/areas.json` index derived from villages.csv + materialised files.

4. **System / engine** (`households`, `household_members`, `sync_log`, live `listings`, refinement tables): **Supabase-managed**; never synced by Claude or edited directly.

### Write-through cache pattern (localStorage → Supabase)

**storage/core.js — the spine.** Every user-state read uses `_get(lsKey, table, fallbackJson, onUpdate)`:
- **Fast path** (cache hit): return localStorage immediately; background `_sbGet` revalidates and fires `onUpdate` if divergent (§18.4 conflict resolution: Supabase wins).
- **No cache** (cold load): `_sbGet` from Supabase; write to localStorage; return.
- **No cache + no Supabase** (true first install): seed from `data/fixtures/*.sample.json` (one-time only).

Every user-state write uses `_save(lsKey, table, value)`: write localStorage immediately (instant render), then fire-and-forget `_sbUpsert` in background. Errors logged + toasted. Both reads and writes go through the core helpers, never directly from page modules.

### Supabase client bootstrap (core.js)

- **Lazy import** of `supabase-client.js` (so the site works before setup). `_initSb()` returns a singleton promise.
- **`household_id` caching** via `_getHid()`: fetches once from `household_members` table using the session user_id, cached for the page lifetime.
- **Auth state invalidation**: `_hid = null` on every `onAuthStateChange` (sign in/out).
- **Safe anon key**: `supabase-client.js` holds only the **publishable key**, designed to be committed and safe iff **every table has Row Level Security** (verified via `list_tables`: all 31 tables RLS-enabled).

### Bidirectional sync contract (CLAUDE.md §18, SUPABASE_SYNC.md §2)

| Trigger | Claude path |
|---------|------------|
| User edits profile in portal | Portal → `storage.js#saveProfile` → localStorage + `_sbUpsert` (fire-and-forget) → Supabase |
| User edits area in Supabase (e.g., research) | Claude via MCP → `mcp__supabase__execute_sql` UPSERT → `tools/sync-areas-from-supabase.mjs` → `tools/build-areas.mjs` → `tests/areas-db-repo-parity.test.js` verifies |
| User adds new house type | Edit `data/house-types.json` → `tools/sync-content-to-supabase.mjs` → `house_types` mirror UPSERT |
| Session start: pull upstream | `node tools/check-supabase-freshness.mjs` → compare snapshot to live MAX(updated_at) → MCP `execute_sql` pulls fresher user-state rows → update snapshot |
| Session end: push changes | Re-SELECT every UPSERT to verify; `tests/supabase-sync.test.js` passes (offline tracked-table check); update snapshot high-water; git commit with "Supabase: N areas, M rows" footer |

**MCP mandatory**: all Supabase DDL via `mcp__supabase__apply_migration` (never direct SQL). Schema migration history is canonical; `schema.sql` is reference only (live shape read via `list_tables`).

### Gaps / skipped areas

- **Household-level multi-device sync**: storage layer doesn't detect conflicts (e.g., user edits profile on phone, Claude edits via MCP simultaneously). ### Known smells / tech debt / risks

### Cache invalidation (medium risk)

`_get` revalidates in the background but fires `onUpdate` async. A fast consumer reads cache, mutates local state, then receives a divergent `onUpdate` after render — can cause flicker or stale renders if the caller doesn't re-render idempotently. Example: profile page reads profile, renders age field; background revalidation arrives with a newer DOB; `onUpdate` fires and page must re-render without flicker. Currently relies on page-level `onUpdate` handlers (e.g., `getProfile({ onUpdate: refresh })`), but not all callers wire them.

### Extend-only constraint + growth (medium risk)

`storage.js` is a 12-line re-export shim (REFACTOR P8); the sibling modules are append-only per §16. Adding a new data type requires a new module (e.g., `storage/new-thing.js`) + export from `storage.js` + Supabase table + MCP migration + `tests/supabase-sync.test.js` entry + snapshot entry. The constraint prevents rewrites but doesn't prevent bloat. Once 40+ functions are exported, the surface becomes hard to reason about.

### Schema drift between schema.sql and migration history (low risk, mitigated)

`supabase/schema.sql` is a **reference snapshot** (not canonical). The live shape is the MCP migration history + `list_tables`. If someone edits schema.sql without applying a migration, the file drifts. Currently caught by code review (schema.sql is not re-generated; it's a manual edit). Recommendation: add a CI check that `schema.sql` and the migration history are in sync, or delete `schema.sql` and rely purely on migration history.

### RLS surface (high risk, currently enforced)

Every table uses RLS; the policy list is fixed and manually verified (via `list_tables` 2026-06-15, all 31 tables RLS-enabled). RLS enforcement is **not tested by the harness** — only via `list_tables` at session start (operator-run MCP call). If an RLS policy is accidentally dropped on the live Supabase project (e.g., via dashboard manual edit), the app continues working but data leaks to any authenticated user. **Mitigation**: mandatory `list_tables` check at session start (CLAUDE.md §8 Step 0); CI/CD that reapplies policies on deploy.

### Error/retry handling in _sbUpsert (medium risk)

`_sbUpsert` is fire-and-forget. Network error = logged to console + toasted to user, but write is lost. Next `_sbGet` will re-fetch from Supabase (not localStorage), but if the user closes the browser before that revalidation, the unsaved change is gone. `_save` writes localStorage first (instant render), so data is locally safe, but Supabase gets a stale view until the user reloads or next session's freshness check pulls the stale row and the user re-saves. Low likelihood (stable network in current deployment) but high blast radius (user data loss).

### Offline behaviour — incomplete (medium risk)

localStorage cache supports offline reads. But there's no offline write queue — `_save` writes localStorage but `_sbUpsert` fails immediately if offline, silently. User doesn't know the change didn't reach Supabase. Next session's freshness check will reveal the Supabase row is stale, but by then the user may have edited on another device and lost work.

### `ask.js` lacks the write-through pattern (low risk, feature-early)

Unlike other user-state, `ask_conversations` are relational (many per household) and written directly to Supabase with no localStorage cache. First load requires a fetch; no offline reads possible. This is acceptable for chat (ephemeral, low data-loss impact) but less resilient than the blob pattern.

### Snapshot `_note` fields get very long (documentation risk)

`sync-state.json` includes migration history in `_note` fields — e.g., the `areas` note is 4000+ chars, recording coordinate corrections, deactivations, re-attributions, etc. over 3 weeks. Useful for archaeology but the file is getting unwieldy. No operational risk (notes are comments), but readability is poor. Consider externalizing old notes to `docs/archive/sync-log/` and keeping only the last 2-3 weeks in the live file.

### Realm of MCP write paths (philosophical risk)

`storage.js` (portal) and MCP (Claude) both write Supabase. Conflict resolution is timestamp-based: fresher row wins (§18.5 user-state always wins Supabase). But if a slow network delays the portal write and Claude writes later, Supabase picks Claude's value. CLAUDE.md §18.4 says "never overwrite a user-edited row unless the user explicitly says overwrite" — but the timestamp-wins rule can't tell intent. **Mitigation**: mandatory freshness check at session start (detect stale rows before editing) + explicit user approval before Claude writes (e.g., "set my LISA cap to £4,000" must be a direct user instruction, not inferred).

### Refactor opportunities (Fable to sequence — note these are SENSITIVE; each is its own §17 phase with the sync ceremony)

### 1. Migrate `storage/*.js` to TypeScript + type-safe getters

**Why**: guard-railed modules are append-only, so the 45-function surface can't shrink. Runtime type errors (e.g., `getProfile()` returns `{ person: null }` instead of `{ person: { name: string } }`) are caught late. TypeScript would surface shape mismatches at compile time.

**Risk**: Refactor P8 was already a large rewrite; moving to TS is another. Requires a build step or esbuild shim.

**Phase scope**: `storage.js` + `storage/*.js` → `.ts` + tsconfig + type definitions for all 45 functions. Tests stay JS. No new capabilities, pure safety.

### 2. Add offline write queue + retry

**Why**: `_save` fires-and-forgets `_sbUpsert`. Network hiccup = lost write (though localStorage is safe). User can't work offline.

**Risk**: Adds state machine (pending queue, ack/retry). Interaction with conflict resolution (fresher row wins — does a queued write lose to a concurrent portal edit?).

**Phase scope**: Add `_pendingWrites` Map in core.js; `_sbUpsert` enqueues failed writes; `_initSb` retries on boot + on network restore. Revalidation after every retry.

### 3. Persist `ask_conversations` to localStorage + revalidate

**Why**: `ask.js` is one-off relational (no write-through cache). Cold load requires fetch; no offline reads.

**Risk**: Conversations are unbounded rows (many per household). localStorage can fill up. Revalidation strategy is complex (last-modified per row, or re-fetch all on open?).

**Phase scope**: `storage/ask.js` gains cache pattern similar to blobs (read localStorage, revalidate in background). Decide on per-row vs bulk revalidation.

### 4. Add generated TypeScript types from Supabase schema

**Why**: Page modules currently use duck typing (`const profile = await getProfile(); profile.person.name`). IDE can't autocomplete; typos found at runtime.

**Risk**: Requires CI to auto-generate types from live schema (or from MCP `list_tables` output). Schema changes would need type regeneration.

**Phase scope**: Set up `mcp__supabase__generate_typescript_types` (or equivalent) in CI/pre-commit; commit generated `types/supabase.d.ts`; update page imports to use the types.

### 5. Split snapshot into versioned changelog + current state

**Why**: `sync-state.json` is now 5000+ lines; `_note` fields are archaeology. Hard to read what the current state is.

**Risk**: Break the single-source-of-truth principle if old and new files diverge.

**Phase scope**: `data/snapshots/sync-state.json` keeps only current high-water marks (100 LOC). Move historical notes to `docs/archive/sync-changelog/YYYY-MM-DD.md`. Update session-start to read only current snapshot.

### 6. Add schema-drift CI check

**Why**: `supabase/schema.sql` is a stale reference if migrations are applied without updating it. Can't tell if the file is canonical or just a snapshot.

**Risk**: If the file is truly reference-only, delete it (source is migrations + live schema). If it should be canonical, the migration history is the source of truth, and the file must be auto-generated from it.

**Phase scope**: Either (a) remove `schema.sql` and add a comment pointing to migrations, or (b) add a CI step that regenerates it from `list_tables` and fails if the regenerated file differs from checked-in.

### 7. Formalize multi-device conflict detection

**Why**: Two devices editing the same user-state row simultaneously: RLS + UPSERT merges, but which write wins is timestamp-based (last-write-wins), not intent-based. User may not notice.

**Risk**: Hard problem; every user-state table would need a version/conflict resolution strategy.

**Phase scope**: Decide on strategy (last-write-wins, operational transform, CRDT, or pre-lock + retry). Document in SUPABASE_SYNC.md. If needed, add a version column to user-state tables.

### How testing works

### Discovery & run model (run-intelligence-tests.mjs)

1. **Import all 63 test files** via dynamic `await import()` — each exports `async function register({ test, assert, assertEqual, fixtures })`.
2. **Load fixture data** (`data/fixtures/{finances,investments,criteria}.sample.json`) → sanitized synthetic household.
3. **Call each `register()`** function, which calls `test(name, fn)` to queue tests. The `test()` function wraps each `fn()` with try/catch, appends `{ name, pass, error? }` to a results array.
4. **Run responsive lint** as a synthetic test: `runResponsiveLint()` → regression check (count-based baseline).
5. **Spawn `supabase-sync.test.js` as a child process** (see below); capture exit code + summary line; add ONE honest `{ name: 'supabase-sync suite (N passed, M failed, O skipped)', pass }` result.
6. **Print summary** (PASS/FAIL per test, totals, exit code 0 iff all pass).

The fixture data is sanitized sample — CLAUDE.md §18.1 defines user-state data as Supabase-only; no real household data in repo.

### Assert & schema helpers

**`tests/assert.js`** (58 lines): exports `test(name, fn)`, `assert(cond, msg)`, `assertEqual(actual, expected, msg)`. Minimal — designed to work in both Node (via `run-intelligence-tests.mjs`) and browsers (via `tests.html` import).

**`tests/schemas.js`** (guard-railed §16): JSON schema validators. Exported: `validateAreaDetail()` (§2 schema per `data/schema/area.schema.json`), profile shape, criteria shape. Used by characterization tests to ensure shape invariants hold before refactoring.

### Test styles: characterization vs unit vs schema

1. **Characterization tests** (5 files: `characterization-*.test.js`): run the full computation chain with synthetic fixtures, assert structural invariants (e.g., "affordability verdict is one of BANDS", "headline is non-empty string", "money flow has 4 buckets"). These are **regression anchors** — if a refactor accidentally moves a computation, these fail first, catching subtle breaks early.

2. **Unit tests** (most of the suite): test individual functions: normalise(), classify(), match(), etc. Assert contract (garbage in → sensible error; valid in → valid out) and **internal consistency** (cheaper never worse than dearer; mutation contract holds; no side effects).

3. **Schema/parity tests** (`supabase-sync.test.js`, `areas-db-repo-parity.test.js`, `docs-consistency.test.js`, `profile-schema.test.js`): assert structural invariants across the system. E.g., `areas-db-repo-parity` rebuilds `data/areas.json` and compares row counts vs DB, catching desync early.

### Offline vs online in supabase-sync.test.js

`tests/supabase-sync.test.js` (~120 lines, imported as child process in CI):

- **Offline checks** (run in CI, deterministic): snapshot validity (file exists, valid JSON, 23 tracked tables listed); shape invariants (debt tables have `count`, data tables have `last_synced_at`); vocabulary checks (REACTIONS, REJECT_REASONS, PERSONAL_STATUSES, baseline gates, purge contract).
- **Online checks** (skipped in CI, no secrets): fetches live Supabase tables via execute_sql, compares row counts + timestamps vs snapshot, validates RLS policies. Claude runs these at session start/end (§8 §18) via MCP.
- **Summary line** printed: e.g., "25 passed, 0 failed, 7 skipped" — harness records ONE honest pass/fail based on exit code, never fabricates per-test lines.

### Responsive lint baseline mechanism

`tools/lint-responsive.mjs` discovers CSS/JS files in `assets/`, parses @media preludes, applies 8 rules (r-no-max-width-media, r-canonical-bp, r-tap-target, etc.). Results are **fingerprints** (rule|file|snippet) → count.

- **Baseline** (`lint-responsive.allow.json`): JSON map of fingerprint → count (currently 6 allowlisted r-no-style-assign).
- **Live check** in harness: if any live count > baseline count, test fails with a regression report.
- **Write baseline** (one-off, manual): `node tools/lint-responsive.mjs --write-baseline` updates the JSON. Only done after deliberate approval (e.g., a new SVG tile that requires dynamic width).
- **Diff invariant**: total violations may only shrink or stay flat; never grow without explicit approval. This prevents lint drift.

### Known smells / tech debt / risks

### Test runtime & CI latency

- `run-intelligence-tests.mjs` is a **sequential** runner — no test parallelism. With 63 suites × fixture import/parse overhead, harness takes ~8–12 seconds on a GitHub runner (measured on ci.yml runs). No measured timeout risk yet, but adding large fixtures or slow tests will compound.
- **Fixture re-read per test file**: each file reads `data/fixtures/*.sample.json` independently (via `deriveFinances()` etc.). Centralizing this would save ~500ms.

### Baseline drift risk

- `lint-responsive.allow.json` is **commit-tracked, not auto-generated**. If a PR adds a new tile with unavoidable r-no-style-assign violations, the baseline must be manually approved. Currently only 6 entries (low friction), but as the codebase grows, this could become noisy. No auto-baseline-bump protection — a careless approval could silently allow regressions.
- **Fingerprint stability** depends on snippet normalization (whitespace collapse). A refactor that reflows CSS comments or splits long selectors will change fingerprints, requiring baseline re-approval even if violations didn't grow.

### Online test gaps (low confidence in deployment)

- **Supabase sync checks skip in CI** (no secrets). Claude runs them at session end, but there's no CI gate if a PR breaks migration history or introduces an RLS policy conflict. The fix would require a CI-accessible test Supabase project (cost + complexity).
- **Listings fetch dry-run not in CI**. The fetcher + normalise pipeline is tested offline (test fixtures), but the live Apify actor interaction is manual-only (workflow_dispatch dry_run = true). A breaking API change in Apify would surface only when the workflow runs.

### Documentation rot risk

- **Instruction docs (CLAUDE.md, DESIGN.md, SUPABASE_SYNC.md) are manually kept in sync**. The `docs-consistency.test.js` guards path mentions and shim claims, but semantic drift can occur (e.g., "the stored-procedures directory exists" when it doesn't anymore). The test catches dead paths, not broken statements.

### Missing coverage

- **No characterization test for listing detail enrichment** (HTML rendering, imagery, descriptions). The `listing-detail.test.js` tests the JS module, but the page-level cascade from API → storage → DOM is not pinned.
- **No end-to-end journey test**: set up a household, save criteria, fetch listings, react to a few, refine, export shortlist, validate the entire flow. Currently each step is tested in isolation.
- **Ask feature (ask-tools, ask-storage)** added recently (June 16). Tests exist but coverage is likely thin — no characterization snapshot yet.

### Flakiness risk

- **Online verify-area-coords.js** runs in resolve-areas.yml (GitHub Actions, open internet). Reverse-geocode APIs (postcodes.io, OpenStreetMap Nominatim) can be slow or rate-limited. Workflow has no retry logic; a flaky API call causes the workflow to fail and leave the branch pending.
- **Concurrency race in fetch-listings.yml**: the once-per-day gate reads workflow history via GitHub API. Under extreme load (e.g., manual dispatch + schedule fire within seconds), a race could allow a double-fetch. Currently mitigated by concurrency { cancel_in_progress: false }, but the gate logic is not atomic.
